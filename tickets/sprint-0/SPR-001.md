# Business Context

The Backend Engineering Simulator is entering the platform foundation phase.

As future modules (User, Product, Inventory, Order, Payment, etc.) are developed, every REST API must return responses in a consistent format.

Without a common response contract:

- Frontend teams need to handle multiple response structures.
- Mobile applications become harder to maintain.
- Error handling becomes inconsistent.
- API documentation becomes difficult to understand.

To solve this problem, the Platform Team has decided to introduce a **Standard API Response Model** that every future REST endpoint will use.

This ticket establishes that contract.

---

# Objective

Design and implement a reusable response wrapper for all future REST APIs.

This ticket should only create the response models.

No controllers or business logic should be implemented.

# Functional Requirements

The response wrapper should support:

## Success Response

Example

```json
{
  "success": true,
  "message": "Request completed successfully",
  "data": {},
  "timestamp": "...",
  "path": "/api/example"
}
```

---

## Error Response

Example

```json
{
  "success": false,
  "message": "Validation failed",
  "error": {
    ...
  },
  "timestamp": "...",
  "path": "/api/example"
}
```

---

## Validation Errors

The response model must support multiple validation errors.

Example

```json
{
  "field": "email",
  "message": "must not be blank"
}
```
